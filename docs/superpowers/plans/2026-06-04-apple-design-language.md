# Apple Design Language

The editorial layer for Apple apps on iOS, iPadOS, and macOS: the parts of "feels Apple-quality" that aren't a spacing number — copy, interaction states, empty/error states, and the judgment of when to follow versus tastefully break the Human Interface Guidelines. Think in capabilities (touch vs. pointer, compact vs. regular vs. window), not device names.

This skill is the **entry point** to the design language. It owns four things and routes everything else.

## HIG is the floor, not the ceiling

The Human Interface Guidelines encode what is **safe and approvable**. Meet that floor everywhere. But Apple's own best apps — and every Apple Design Award winner — exceed or selectively break the defaults where it serves the user. "HIG-compliant" and "great" are not the same target. Your job: clear the floor without exception, then raise the ceiling deliberately.

### Deviation decision gates

Deviate from a HIG default only when **all four** hold:

1. **Named benefit** — you can state the user benefit in one sentence that isn't "looks better," "feels modern," or "cleaner."
2. **Survives the slop check** — it is not one of the numbered tells below (and will pass `swiftui-design-check` once that exists).
3. **Survives assistive tech** — still correct under VoiceOver and Dynamic Type at AX5 (see `ios-accessibility`).
4. **Systematic** — applied as a rule across the app, not a one-off that creates an inconsistency.

If any gate fails, use the HIG default. Deviation is earned, not assumed.

## Anti-slop tells (numbered — the contract with `swiftui-design-check`)

These are the Apple-native equivalents of web "AI slop." Each is numbered so the `swiftui-design-check` linter (sub-project #2) can map a scanner rule to it. **Statically detectable** ones are marked ⚙.

1. ⚙ **System-gray everything** — flat `.gray`/`Color(.systemGray)` fills where a system material or hierarchical foreground style belongs.
2. **Sheet-for-everything** — a modal `.sheet` where a push, popover, menu, or inline disclosure fits the navigation depth better.
3. ⚙ **Unlabeled icon-only buttons** — an icon `Button`/`Label(systemImage:)` with no `accessibilityLabel` and no visible text.
4. ⚙ **Untouched `.automatic` styling** — default button/list/navigation styling shipped with no deliberate choice behind it.
5. ⚙ **Center-everything layout** — no leading-edge alignment hierarchy; titles, body, and controls all centered.
6. ⚙ **Hardcoded font sizes** — `.font(.system(size: 17))` instead of a semantic text style.
7. ⚙ **Twin full-width CTAs** — primary and secondary actions given equal visual weight, destroying hierarchy.
8. ⚙ **Emoji as iconography** — emoji where an SF Symbol is the native choice.
9. ⚙ **Placeholder-as-label** — relying on `TextField` placeholder text instead of a persistent field label.
10. **Spinner-for-everything** — indeterminate spinners where a skeleton or determinate progress fits, with no empty/error variant designed.
11. **Toast for the critical** — a banner/toast where an `.alert` is required (data loss, destructive confirmation).
12. ⚙ **Gradient-and-glow slop** — decorative purple-ish gradients, drop-glows, or faux depth not derived from system materials.
13. **Generic empty state** — "No items" with no cause and no next action (see `references/empty-error-states.md`).
14. ⚙ **Title Case body copy** — title-casing sentences, labels, or descriptions that should be sentence case (see `references/ux-writing.md`).

## Quick reference

### Capitalization

| Use | Case |
|---|---|
| Nav-bar titles, buttons, menu items, alert titles | Title Case |
| Labels, body text, alert messages, hints, footnotes, placeholders | Sentence case |

### Button verbs (pick the precise word)

| Word | Use for |
|---|---|
| Done | Dismiss, keeping changes already applied |
| Save | Commit changes that aren't applied yet |
| Cancel | Dismiss, discarding changes |
| OK | Acknowledge an alert with no alternative |
| Delete / Remove | Destructive — pair with confirmation; never "OK" |

### Empty/error: which surface

Data loss or destructive confirmation → `.alert`. Recoverable, non-blocking → inline state or banner. No data yet → designed empty state with cause + action. Full tree in `references/empty-error-states.md`.

- [ ] **Step 1: Write the file.** It MUST contain these sections, each with concrete Apple-native guidance (not generic copywriting). Use the seed content below verbatim as the spine and expand each bullet into 1–3 sentences of guidance with a good/bad example pair where marked.

Required sections and content:

1. **Voice & tone** — Apple voice: clear, direct, human, never cute-for-cute's-sake. Second person ("you"), active voice, present tense. No marketing fluff in functional copy.
2. **Capitalization** — restate the table from the hub and add the macOS nuance: macOS **menu items use Title Case**; iOS/iPadOS body and labels use Sentence case. Include good/bad pair: ✅ "Move to trash" (button is Title: "Move to Trash"; sentence label: "This moves the file to the trash.").
3. **Buttons & labels** — verbs name the outcome, not the mechanism. ✅ "Save" / ❌ "Submit". Avoid "OK" except in unavoidable alerts. Never label a destructive button "OK". One primary verb per screen.
4. **Error messages** — three-part structure: what happened + why (plainly) + how to fix. No error codes, no jargon, no blame. ✅ "Couldn't save the note. You're offline — it'll sync when you reconnect." / ❌ "Error -1009". Tie to anti-slop tell #14 and to `empty-error-states.md`.
5. **Alerts** — title is a Title-Case noun-phrase or question; message is a sentence-case explanation; buttons are verbs. Destructive button uses `.destructive` role + the real verb ("Delete").
6. **Notifications & permission prompts** — say *why* before the system prompt (pre-permission priming); the `Info.plist` usage string is sentence case, specific, and benefit-led. ✅ "Lets you attach photos to entries." / ❌ "This app needs photo access."
7. **Empty-state copy** — name the cause and the next action; never a dead "No items." (cross-link `empty-error-states.md`).
8. **macOS deltas** — Title-Case menus, sentence-case everything else; menu items use ellipsis "…" when the command opens further UI; Mac copy can assume pointer + keyboard ("Click", "⌘S") where iOS says "Tap".

End with a one-line link back: `See the hub: ../SKILL.md`.

### Task 3: `references/interaction-states.md`

**Files:**
- Create: `src/skills/apple-design-language/references/interaction-states.md`

- [ ] **Step 1: Write the file** with these required sections:

1. **The five-state discipline** — every interactive control must define: default, pressed, focused, disabled, and (pointer platforms) hover. Hover/focus are the macOS + iPad-with-pointer deltas; never ship a Mac/iPad control with no hover or focus treatment. Disabled ≠ hidden — hide only when the control is irrelevant in context; disable (with a reason available) when it's temporarily unavailable.
2. **Forms — structure** — group related fields (`Form`/`Section`); one idea per section; persistent labels (never placeholder-as-label, anti-slop #9); logical field order; keyboard type matches content (`.keyboardType`, `.textContentType` for autofill).
3. **Forms — validation timing** — validate on submit for required/format errors; validate inline only for things the user can fix as they type (e.g., password rules), and only *after* first blur, never on every keystroke from empty. Show the error adjacent to the field, in sentence case, saying how to fix it.
4. **Required vs optional** — mark the *minority*. If most fields are required, mark "Optional"; if most are optional, mark required. Don't asterisk everything.
5. **Focus order** — define `@FocusState` order for keyboard/return-key advance on iOS; ensure tab order is sane on macOS/iPad; the Return/Next key should move to the next logical field and submit on the last.
6. **Loading patterns** — choose by knowledge of duration and layout: determinate progress when you know the fraction; skeleton/redacted placeholder when you know the layout and it's >~1s; spinner only for short, unpredictable waits with no known layout. Anti-slop #10: never ship spinner-only with no empty/error variant. Avoid layout shift when content arrives.
7. **Destructive actions** — confirm via `.alert` with a `.destructive`-role button named with the real verb; provide undo where feasible (better than a confirardless prompt). Never a toast for destructive confirmation (anti-slop #11).
8. **Disabled affordances** — a disabled primary button must let the user discover *why* (helper text, or enable + explain on tap). Silent dead buttons are a slop tell.

End with: `See the hub: ../SKILL.md`.

### Task 4: `references/empty-error-states.md`

**Files:**
- Create: `src/skills/apple-design-language/references/empty-error-states.md`

- [ ] **Step 1: Write the file.** Lead with a decision tree, then one subsection per state. Required content:

**Decision tree (verbatim spine, expand each leaf):**
```
Is there content to show?
├─ No, and the user hasn't acted yet        → FIRST-RUN empty state
├─ No, because this collection is genuinely empty → NO-DATA empty state
├─ No, because a search/filter matched nothing    → NO-RESULTS state
├─ No, because we can't reach the network          → OFFLINE state
├─ No, because the user denied a permission         → PERMISSION-DENIED state
└─ No, because an operation failed                  → ERROR/FAILURE state
```

For each of the six states, specify: **what to show**, **what to say**, **what action to offer**:

1. **First-run** — explain the value in one line + a primary action to create the first item. Not a marketing screen. ✅ "Your trips will appear here. Add your first flight." + [Add Flight].
2. **No-data** — like first-run but the user has used the app; lighter, still offers the create action.
3. **No-results** — confirm the query, suggest broadening; offer "Clear filters". Never identical to no-data (different cause, different fix).
4. **Offline** — distinguish "you're offline" from "the server failed"; show cached content if available; offer Retry; don't destroy unsynced work.
5. **Permission-denied** — explain what the feature needs and why, and route to Settings (`UIApplication.openSettingsURLString`); never nag repeatedly; provide a degraded-but-usable path if possible.
6. **Error/failure** — three-part copy (what/why/fix, per `ux-writing.md`); Retry for transient; for data loss use `.alert`; preserve the user's input. Tie to anti-slop #11 and #13.

Add a short **"Anatomy of a good empty state"** note: SF Symbol or simple illustration (not emoji), a sentence-case line of copy, one primary action; respects Dynamic Type and dark mode; centered is acceptable here (the one place center-everything isn't a slop tell).
